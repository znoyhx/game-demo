# 二、产品概述

## 2.1 产品一句话定义

PixelForge Agent 是一个以**精美像素风**为视觉载体、基于**多 Agent 协作**构建的 AI 游戏平台。用户可以通过自然语言快速创建并进入一个可玩的冒险世界，系统自动生成地图、剧情、任务、NPC、敌人策略与动态事件，并支持**自动存档、自动读档、可解释交互、模块化扩展**。

## 2.2 产品愿景

重新定义“玩中学、学中创”的边界，让用户不仅是玩家，也是世界共创者。
通过多 Agent 系统，让游戏具备以下能力：

* 会生成世界
* 会驱动角色
* 会根据玩家行为变化
* 会解释自己的决策
* 会在每次启动时延续上一次的世界状态

## 2.3 产品定位

本产品不是单纯的 NPC 对话工具，也不是单纯的地图生成器，而是一个具备如下特征的**游戏 Agent 平台**：

1. **像素风精美游戏体验**
2. **多 Agent 协同生成内容**
3. **动态 NPC 与智能敌人**
4. **强状态管理与强可测试性**
5. **自动存档与持久世界**
6. **高解耦、高扩展的系统设计**

# 三、立项背景与机会分析

## 3.1 背景

目前大多数 AI 游戏类项目存在以下问题：

1. 偏“聊天机器人化”，缺少真正的游戏系统
2. 关卡、任务、NPC 常为静态内容，互动深度弱
3. 策略对抗能力不足，缺少实时或准实时应变
4. 演示效果单薄，缺乏可持续游玩闭环
5. 测试困难，很多功能必须依赖冗长前置流程
6. 缺少状态持久化，无法形成“持续世界”

## 3.2 机会点

本赛题鼓励开发能够参与：

* 游戏内容生成
* 关卡设计
* NPC 行为逻辑
* 实时策略对抗

因此，最佳解法不是做单一 Agent，而是做一个**多模块、多 Agent、可持续运行的游戏系统**。

PixelForge Agent 将通过“像素风世界 + 多 Agent 内核 + 持久存档机制”同时满足：

* 技术复杂度高
* 演示效果强
* 功能丰富
* 可测试性强
* 扩展空间大

# 四、产品目标

## 4.1 核心目标

打造一个支持以下能力的可玩平台：

1. 用户一句话生成像素风游戏世界
2. 游戏包含地图、任务、NPC、敌人、动态事件
3. 系统能根据玩家行为调整叙事与策略
4. 每次启动自动加载存档，延续世界状态
5. 各功能模块支持独立测试，无需依赖完整通关链路

## 4.2 比赛目标映射

### 技术维度（50%）

#### 创新性（20%）

* 多 Agent 协同：世界、任务、关卡、NPC、敌人、导演、玩家建模、解释
* 将“内容生成”与“策略对抗”放到同一系统中
* 将“视觉像素游戏”与“可解释 AI”融合

#### 效率（15%）

* 网站形态便于快速部署
* 状态机与规则引擎控制核心逻辑
* 生成与渲染解耦，支持缓存与局部刷新

#### 鲁棒性（15%）

* 状态合法性校验
* 生成结果结构化约束
* 回退机制与自动重试
* 存档恢复与异常兜底

### 应用维度（50%）

#### 实用价值（20%）

* 可用于娱乐
* 可用于教育闯关
* 可用于创意写作与独立游戏原型
* 可用于策略训练/推理训练

#### 人机交互（15%）

* 自然语言建世界
* 像素风视觉界面
* NPC 可解释对话
* 复盘与建议反馈

#### 展示效果（15%）

* 现场一句话创建世界
* 进入游戏试玩
* 展示 NPC 互动、敌人策略变化、自动读档续玩
* 全流程完整且流畅

# 六、产品形态决策

## 6.1 为什么选择 Web 作为主形态

### 原因 1：更适合比赛演示

* 打开浏览器即可展示
* 无需现场安装复杂环境
* 适合大屏投屏
* 更容易保证演示流畅度

### 原因 2：测试成本低

* 前后端接口更易 Mock
* 模块功能可单独测试
* 页面状态可快速重置
* 更容易做测试面板和开发者工具页

### 原因 3：存档与自动加载更方便

* 可本地存储 LocalStorage / IndexedDB
* 可扩展云端用户存档
* 每次启动自动恢复世界状态容易实现

### 原因 4：扩展性更强

* 后续可打包成 exe
* 便于接入多模态模型、可视化编辑器、创作工具

## 6.2 后续桌面封装策略

在比赛答辩中可说明：

* 当前主交付为 Web
* 后续通过 Tauri/Electron 打包为 Windows 桌面客户端
* 保持业务层与渲染层不变，仅更换运行容器

这能体现架构成熟度。

# 八、核心设计原则

## 8.1 强可测试性

任何主要功能都应支持**无前置大流程测试**。
例如：

* 不需要先完成三章主线才能测试 Boss AI
* 不需要完整创建世界才能测试 NPC 模块
* 不需要正式进入游戏才能测试存档机制

## 8.2 高解耦

* 世界生成、剧情生成、NPC 行为、战斗策略、存档系统分离
* UI 层不直接耦合模型逻辑
* 渲染层不直接操作业务规则
* 数据层统一通过状态中心管理

## 8.3 强扩展性

* 可新增职业、地图、NPC、敌人类型
* 可切换模型能力
* 可拓展教育模式与创作模式
* 可后续加联机或多模态

## 8.4 持久化优先

* 自动存档
* 自动读取最近一次存档
* 关键状态变更即写盘
* 存档可回滚

## 8.5 展示优先

* 画面精美
* 信息布局清晰
* 关键 AI 行为可被看见
* 关键智能决策可被解释

# 九、系统角色与多 Agent 设计

本产品采用多 Agent 协作架构，至少包含以下 8 个智能体。

## 9.1 World Architect Agent

职责：

* 生成世界观与区域结构
* 设定阵营、资源、时代背景
* 输出游戏初始规则

输入：

* 用户主题、风格、游戏目标

输出：

* 世界设定
* 地图区域
* 阵营设定
* 世界事件池

---

## 9.2 Quest Designer Agent

职责：

* 生成主线、支线、隐藏任务
* 设定奖励、前置条件、任务分支

输出：

* 任务树
* 触发条件
* 奖励规则
* 支线关联关系

---

## 9.3 Level Builder Agent

职责：

* 生成区域布局
* 配置障碍物、传送点、道具点、战斗点
* 控制节奏曲线

输出：

* 地图逻辑数据
* 交互点数据
* 区域难度分布

---

## 9.4 NPC Brain Agent

职责：

* 驱动 NPC 对话与行为
* 管理 NPC 记忆、立场、情绪、秘密
* 根据玩家行为动态调整互动内容

输出：

* 对话内容
* 态度变化
* 行为建议
* 信息披露决策

---

## 9.5 Enemy Tactician Agent

职责：

* 控制敌人和 Boss 的策略行为
* 根据玩家行为模式调整对抗策略
* 提供准实时反制

输出：

* 行动决策
* 策略切换
* 难度调整参数

---

## 9.6 Game Master Agent

职责：

* 统筹游戏节奏
* 触发突发事件
* 调节资源与张力
* 防止过于线性或过于卡关

输出：

* 动态事件
* 剧情推进建议
* 世界扰动

---

## 9.7 Player Model Agent

职责：

* 分析玩家行为风格
* 判断玩家属于探索型、战斗型、剧情型、速通型等
* 为难度调整与引导提供依据

输出：

* 玩家画像
* 卡点识别
* 风险预测

---

## 9.8 Explain & Coach Agent

职责：

* 给出可解释反馈
* 输出策略复盘
* 提供引导与建议
* 在教育模式下做知识点总结

输出：

* 决策解释
* 失败原因
* 复盘报告
* 行为建议

# 十一、详细功能需求

## 11.1 游戏创建模块

### 功能目标

用户通过少量输入快速生成一个像素风世界。

### 输入项

* 游戏主题
* 世界风格
* 难度等级
* 游戏目标
* 学习目标（可选）
* 是否偏剧情 / 探索 / 战斗

### 输出项

* 世界名称
* 地图区域
* 核心阵营
* 主线任务
* 初始 NPC 集合
* 初始资源配置
* 初始剧情设定

### 需求点

1. 创建流程控制在 1 分钟内
2. 支持一键生成默认方案
3. 支持预设模板
4. 支持“快速试玩模式”
5. 支持“开发测试模式”

### 可测试性要求

* 支持直接加载预设世界
* 支持跳过完整创建流程
* 支持用 mock 配置生成固定地图和固定 NPC

---

## 11.2 像素世界渲染模块

### 功能目标

用精美像素风呈现游戏世界，提高视觉完成度和现场表现力。

### 画面风格要求

* 统一像素风 UI
* 具有层次感的地块、建筑、光影
* 角色立绘/头像为像素风
* 事件和交互有明显反馈
* 色彩风格统一，避免廉价感

### 画面组成

* 顶部：世界名、当前区域、时间/天气
* 左侧：地图区域或小地图
* 中央：主画面/场景区
* 右侧：任务、背包、关系、状态信息
* 底部：对话框、操作栏、日志

### 需求点

1. 支持像素地图渲染
2. 支持 NPC 图标与交互提示
3. 支持关键点高亮
4. 支持战斗/事件动画反馈
5. 支持状态变化浮层提示

### 可测试性要求

* 地图渲染可脱离 Agent 独立调试
* UI 组件可 Storybook 化或单独预览
* 任意区域可单独加载渲染

---

## 11.3 地图与区域模块

### 功能目标

构建由多个像素区域组成的可探索世界。

### 数据结构

每个区域应包含：

* 区域 ID
* 名称
* 类型
* 难度等级
* 可进入条件
* 交互点列表
* NPC 列表
* 敌人生成规则
* 事件池
* 资源点

### 基础能力

1. 至少 3 个初始区域
2. 支持区域切换
3. 支持区域锁定与解锁
4. 支持隐藏区域
5. 支持传送点
6. 支持环境状态变化

### 可测试性要求

* 任意区域支持直接跳转
* 支持通过调试菜单解锁区域
* 支持单独测试地图进入/退出逻辑

---

## 11.4 任务系统模块

### 功能目标

为玩家提供清晰目标与分支路径，增强结构化游玩体验。

### 任务类型

* 主线任务
* 支线任务
* 隐藏任务
* 教学任务
* 动态任务

### 任务字段

* 任务 ID
* 类型
* 标题
* 描述
* 触发条件
* 完成条件
* 奖励
* 失败条件
* 依赖关系
* 分支结果

### 需求点

1. 至少 1 条主线
2. 至少 3 条支线
3. 支持多分支结果
4. 支持动态触发任务
5. 支持任务日志记录
6. 支持任务状态追踪

### 可测试性要求

* 支持直接激活指定任务
* 支持直接完成/失败任务
* 支持任务条件模拟
* 不要求玩家从头推进主线才能验证后续逻辑

---

## 11.5 NPC 智能交互模块

### 功能目标

让 NPC 成为真正有状态、有记忆、有立场的智能角色。

### NPC 数据结构

* NPC ID
* 名称
* 职业/身份
* 阵营
* 性格标签
* 情绪状态
* 记忆摘要
* 对玩家关系值
* 信任值
* 可透露信息
* 隐藏秘密
* 当前目标

### 交互能力

1. 多轮对话
2. 根据玩家历史改变说法
3. 根据信任度决定信息开放程度
4. 支持任务发放
5. 支持道具交换
6. 支持立场变化
7. 支持关系网络变化

### 可解释性需求

* 显示 NPC 当前态度
* 展示关键关系变化原因
* 在调试模式下展示简要决策依据

### 可测试性要求

* 支持直接打开任意 NPC 对话页
* 支持修改关系值测试不同分支
* 支持注入记忆状态测试逻辑一致性

---

## 11.6 战斗与敌人策略模块

### 功能目标

体现“实时策略对抗”的赛题价值，让敌人具备适应性。

### 基础模式

建议 MVP 采用：

* 轻量回合制
  或
* 半实时指令式对抗

### 敌人策略类型

* 激进压制
* 防守消耗
* 诱导陷阱
* 召唤支援
* 资源封锁
* 反制玩家常用套路

### 决策输入

* 玩家血量/资源
* 玩家常用动作
* 地图环境状态
* 战斗轮次
* Boss 阶段
* 玩家画像标签

### 需求点

1. 至少 3 种策略可切换
2. 玩家不同行为触发不同敌人应对
3. Boss 至少具备阶段切换
4. 支持战斗日志记录
5. 支持战斗复盘

### 可测试性要求

* 支持直接进入指定战斗
* 支持模拟玩家行为模式
* 支持固定随机种子复现实验
* 支持一键切换敌方策略验证表现

---

## 11.7 动态事件与导演模块

### 功能目标

让游戏世界具备“活性”，增强戏剧性与观赏性。

### 事件类型

* 天气变化
* 资源减少
* NPC 突发移动
* 阵营冲突
* 隐藏线索暴露
* Boss 提前现身
* 商店价格变化
* 区域封锁/开放

### 触发机制

* 时间触发
* 地点触发
* 任务触发
* 关系触发
* 玩家画像触发
* 难度平衡触发

### 可测试性要求

* 事件可手动触发
* 支持禁用随机以稳定测试
* 支持事件回放

---

## 11.8 玩家画像模块

### 功能目标

识别玩家风格，用于个性化调整。

### 画像标签示例

* 探索型
* 战斗型
* 社交型
* 剧情型
* 速通型
* 谨慎型
* 冒险型

### 输出用途

* 难度调整
* 提示策略
* NPC 反应变化
* 战斗反制
* 复盘展示

### 可测试性要求

* 支持直接写入玩家标签
* 支持通过行为回放生成画像

---

## 11.9 解释与复盘模块

### 功能目标

提升可解释性与展示效果，让评委和玩家都能理解 Agent 的“智能”。

### 输出内容

* 当前局玩家画像
* 关键任务分支变化原因
* NPC 态度变化原因
* 敌人策略变化原因
* 玩家失败或成功关键因素

### 自动加载策略

系统启动时执行：

1. 检测本地是否存在最近有效存档

3. 若不存在，则进入新建世界/选择模板页
4. 若存档损坏，则提示恢复最近可用备份

### 自动存档触发时机

* 区域切换后
* 任务状态变化后
* 战斗结束后
* 关键对话结束后
* 配置变更后

* 定时存档

* 当前任务进度
* NPC 状态与关系值
* 玩家背包与属性
* 战斗状态摘要
* 事件触发记录
* 玩家画像
* 时间线信息

# 十六、解耦设计要求

## 16.1 模块边界

必须拆分为以下层次：

1. **UI 渲染层**
2. **交互控制层**
3. **游戏状态层**
4. **Agent 调度层**
5. **规则引擎层**
6. **持久化层**

## 16.2 解耦原则

* UI 不直接改写核心状态
* Agent 不直接控制渲染逻辑
* 存档模块不依赖具体 UI 结构
* 战斗逻辑与 NPC 逻辑独立
* 地图系统与任务系统通过事件总线通信

## 16.3 推荐通信方式

* 统一状态仓库
* 事件总线/Event Bus
* 结构化 JSON 输出
* Agent 响应 schema 校验

# 二十一、风险与对策

## 风险 1：功能过多导致无法落地

对策：

* 严格控制 MVP
* 优先做“可见智能”闭环
* 非核心功能写入后续规划

## 风险 2：AI 输出不稳定

对策：

* 关键输出结构化
* schema 校验
* 回退模板兜底
* 缓存稳定结果

## 风险 3：现场演示卡顿

对策：

* 预热模型
* 缓存预置世界
* 提供离线测试存档
* 大模型超时走预设兜底

## 风险 4：测试成本过高

对策：

* 建立 debug 模式
* 每模块支持独立注入状态
* 支持跳关和直进场景
