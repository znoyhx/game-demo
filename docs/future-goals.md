## FG-001 强化 Phaser 像素场景表现

- 状态：等待用户指令
- 优先级：高
- 来源：像素场景基础修复完成后的后续增强建议

### 目标

在不破坏现有架构边界的前提下，进一步增强主场景的像素冒险观感，让核心玩法区域更接近真正的 2D 像素 RPG 演示效果，而不是只具备基础可运行能力的占位场景。

### 计划方向

- 增强地表、道路、障碍、装饰等图层表现，提高不同区域的像素辨识度
- 提升玩家移动反馈，包括脚下阴影、移动节奏、朝向切换与稳定的行走表现
- 增强人物与交互点的在场感，例如头顶提示、聚焦高亮与临近交互反馈
- 强化场景氛围元素，例如更稳定的环境特效与区域特色视觉提示
- 为后续真实美术资源替换保留清晰接口，不把表现逻辑耦合进规则、状态、控制器或持久化层

### 约束

- 仅在渲染层与渲染适配层内推进
- 不把业务规则搬进 Phaser 场景
- 不改变现有保存 / 加载契约
- 不移除当前可测试、可注入、可降级的调试路径

### 执行条件

- 只有当用户明确要求开始该项时，才进入实现阶段

## FG-002 场景内敌人显式可视化

- 状态：等待用户指令
- 优先级：高
- 来源：战斗演示可读性提升建议

### 目标

在地图探索与战斗入口之间建立更直接的视觉连接，让玩家在进入战斗前就能清晰知道“正在和谁战斗”，减少只能依赖侧边列表或底部面板理解战局的负担。

### 计划方向

- 在 `GameSceneViewport` / Phaser 场景中为可战斗目标增加显式敌人标识
- 为敌方单位增加名称、小型生命条、危险等级或首领标识
- 让探索信号、敌人实体、战斗入口三者保持一致映射，避免 UI 与规则脱节
- 在首领区域中突出当前可交战目标，减少“地图上看不到敌人”的问题
- 与现有 `combat encounter`、`exploration signal`、`review payload` 契约保持对齐，必要时只扩展渲染所需的视图模型字段

### 约束

- 首领逻辑、战斗合法性与阶段切换仍保留在规则层
- UI 只读取视图模型或控制器暴露的数据，不直接拼接战斗规则
- 不破坏当前调试入口、固定种子与强制战术 / 阶段测试能力
- 所有新玩家可见文案继续保持简体中文

### 执行条件

- 只有当用户明确要求开始该项时，才进入实现阶段

## FG-003 UI 整体整洁度与信息架构重构

- 状态：等待用户指令
- 优先级：最高
- 来源：2026-03-26 用户直接反馈

### 目标

针对当前页面“很丑、极其不整齐、页面繁杂”的问题，重做整体 UI 结构、信息密度、视觉节奏与组件对齐规则，让主流程页面真正符合 PRD 中“可演示、清晰、像素幻想冒险感”的要求，而不是继续停留在功能堆叠式面板状态。

### 计划方向

- 统一主页面、调试页、复盘页的栅格、间距、标题层级、按钮尺寸、卡片边界与对齐规则
- 减少同时出现的面板数量，明确主次层级，避免信息拥挤和视觉噪声
- 重做 HUD、侧栏、日志区、调试区与复盘区的信息编排，优先保证演示时一眼可读
- 将“像素幻想冒险 UI”作为主视觉方向，而不是接近后台面板或开发工具面板的观感
- 为后续视觉验收建立更明确的标准，例如：整齐度、阅读路径、模块密度、单屏聚焦度、演示截图质量

### 约束

- 不把业务规则重新塞回 UI
- 不破坏控制器 / 状态 / 规则 / 持久化边界
- 所有玩家可见文案继续保持简体中文

### 执行条件

- 只有当用户明确要求开始该项时，才进入实现阶段

## FG-004 按 PRD 接入真实 Agent 工作路径

- 状态：等待用户指令
- 优先级：最高
- 来源：2026-03-26 用户直接反馈

### 目标

解决“完全没有接入 PRD 中那几个 Agent、现在几乎全是 mock”的问题。后续需要让 PRD 中定义的关键 Agent 至少具备真实接入路径、切换机制、失败回退和可调试日志，而不是长期只有 mock 演示层。

### 计划方向

- 逐个梳理 PRD 中的 World Architect、Quest Designer、NPC Brain、Enemy Tactician、Game Master、Player Model、Explain & Coach 等 Agent
- 为每个 Agent 明确 mock、真实实现、配置、开关、回退、日志和调试注入方式
- 先保证“可接真实模型但不会破坏 demo”的安全接入方式，再决定哪些模块默认启用真实 Agent
- 保持 schema 校验、可测试性和 deterministic fallback，不允许模型输出直接裸写入游戏状态
- 在 debug / review 中明确展示当前使用的是 mock 还是真实 Agent，以便演示和排障

### 约束

- 不能牺牲当前可测试性
- 不能取消 mock 路径
- 真实 Agent 接入必须遵守 PRD 与架构的模块边界

### 执行条件

- 只有当用户明确要求开始该项时，才进入实现阶段

## FG-005 16-bit 像素风格与像素质量升级

- 状态：等待用户指令
- 优先级：最高
- 来源：2026-03-26 用户直接反馈

### 目标

解决“像素水平低、达不到 16-bit 像素风格”的问题，把当前偏占位、偏低保真的像素表现，升级为更接近 16-bit 像素 RPG 的统一风格，包括场景、角色、UI 装饰、色板、轮廓、层次与动画节奏。

### 计划方向

- 制定 16-bit 风格基线：分辨率密度、色板数量、明暗层级、描边风格、材质语言、装饰规则
- 提升地图地表、建筑、遮挡、前景、阴影和交互点的像素一致性
- 提升角色与敌人的像素表现、站位可读性、动画切换与战斗反馈
- 让 UI 边框、按钮、徽章、面板、标题条和提示元素都统一到 16-bit 像素语义，而不是混合现代扁平样式
- 明确“像素风格验收截图”与“演示镜头标准”，避免后续继续只做功能不做风格质量

### 约束

- 渲染升级不能破坏现有测试入口与调试入口
- 需要优先服务主演示路径，而不是先做大而散的美术扩张
- 所有风格升级都应保持后续资源替换能力

### 执行条件

- 只有当用户明确要求开始该项时，才进入实现阶段
